# TrainLog — Стандарты предпродакшена

> Единый источник правды: UX-система, слой данных, загрузка, ошибки.  
> Все новые экраны и доработки должны следовать этому документу.  
> Документ обновляется по мере согласования решений.

---

## 1. UX / Design System

**Принцип:** Всё переиспользуется. Кнопки, списки, иконки, заголовки, шрифты, цвета и размеры задаются через общие компоненты и токены. Не дублировать разметку и стили по экранам.

### 1.1 Где лежит дизайн

| Что | Где |
|-----|-----|
| Цвета, размеры, шрифты, отступы | `Core/Design/Design.swift` (`AppDesign`, `AppTheme`) |
| Кнопки (основная, строка-действие, «Добавить») | `Core/Design/ButtonComponents.swift` |
| Карточки, блоки настроек, сегменты | `Core/Design/SettingsComponents.swift` |
| Заглушки «нет данных», загрузка | `Design.swift` — `EmptyStateView`, `LoadingView`, `LoadingBlockView` |

**Токены AppDesign:** `sectionSpacing`, `cardPadding`, `rowSpacing`, `blockSpacing`, `cornerRadius`, `primaryButtonHeight`, `primaryButtonColor`, `profileAccent`, `profileAccentOpacity`, `destructiveBackgroundOpacity`, `loadingSpacing`, `loadingMessageFont`, `loadingScale`, `emptyState*`.

### 1.2 Что использовать обязательно

- **Основная кнопка (Войти, Сохранить, Создать, Обновить):** `PrimaryActionButton` + `PrimaryButtonStyle`. Высота и цвет — из `AppDesign`.
- **Строка в блоке профиля/списка (иконка + заголовок + значение):** `ActionBlockRow`. Обёртка блока — `.actionBlockStyle()`.
- **Кнопка «Добавить что-то» в списке:** `AddActionRow` внутри `Button` или `NavigationLink`.
- **Карточка с заголовком и контентом:** `SettingsCard(title:content:)`.
- **Деструктивное действие (удалить, отвязать):** `AccountManagementCard` или кнопка с `role: .destructive`.
- **Пустое состояние:** `EmptyStateView` или `ContentUnavailableView` с единым стилем (иконка, заголовок, описание).

### 1.3 Запрещено

- Хардкодить цвета/размеры/шрифты в экранах вместо использования `AppDesign` и готовых компонентов.
- Создавать «свои» кнопки или строки списка, если уже есть компонент в `Core/Design`.

### 1.4 Идеи на будущее

- Вынести все токены (цвета, отступы, размеры шрифтов, радиусы) в один файл/перечисление для единообразия и темизации.
- Иконки — единый набор (SF Symbols или свой), описанный в одном месте.

## 2. Слой данных / Backend

**Принцип:** Работа с сервером изолирована за протоколами сервисов. Вся сетевая логика — в сервисах (Firestore/другой backend). Экраны вызывают только методы сервисов и не знают деталей Firebase/HTTP. Это позволит в будущем заменить бэкенд (отдельный API) без переписывания экранов.

### 2.1 Текущая структура

- **Протоколы:** `Core/Services/*.swift` (например `ProfileServiceProtocol`, `MeasurementServiceProtocol`, `VisitServiceProtocol`).
- **Реализации:** `Firestore*Service` — работа с Firebase; `Mock*Service` — для превью/тестов.
- **Внедрение:** сервисы передаются через Environment или инициализаторы экранов (DI).

### 2.2 Правила

- Экраны и ViewModels не импортируют Firebase. Они зависят только от протоколов.
- Все вызовы к «серверу» идут через методы сервисов. Не дублировать логику запросов в нескольких местах.
- Ошибки возвращаются через `throws` или callback; отображение ошибок — через единый механизм (см. раздел 4).

### 2.3 Подготовка к отдельному бэкенду

- Сохранять чёткое разделение: **протокол сервиса** ↔ **реализация (Firestore / будущий REST)**.
- Имена методов и доменные модели (Profile, Measurement, Visit и т.д.) не привязаны к Firestore. При переходе на REST меняется только реализация сервиса и, при необходимости, маппинг ошибок.

## 3. Состояние загрузки (Loading)

**Принцип:** Любой экран или переход, где есть загрузка данных с сервера (или тяжёлый подсчёт), обязан показывать явное состояние загрузки. Не должно быть ситуации «открылась страница с заглушкой (нет замеров), через секунду данные подгрузились и появились» без индикатора.

### 3.1 Унифицированный вид загрузки

- **Полноэкранная загрузка списка/экрана:** компонент `LoadingView(message:)` из `Core/Design/Design.swift` — спиннер + текст по центру. Использовать везде вместо дублирования `VStack(spacing: 12) { ProgressView()... }`.
- **Загрузка внутри контента (под кнопкой, не на весь экран):** компонент `LoadingBlockView(message:)` — тот же вид с ограниченной высотой (`minHeight: 200`).
- **Кнопка с запросом:** `PrimaryActionButton(isLoading: true)` — внутри кнопки спиннер, текст скрыт.

### 3.2 Правила

- Перед первым показом данных с сервера: всегда `isLoading = true` → показать лоадер → после ответа `isLoading = false` → показать контент или пустое состояние.
- Pull-to-refresh может показывать индикатор в navigation bar или сверху списка; повторная загрузка не должна оставлять экран без индикатора, если старые данные скрыты.

### 3.3 Где уже есть загрузка (для сверки)

- Выбор профиля, карточка клиента, посещения/абонементы, формы (логин, регистрация, добавить замер/цель/подопечного). Везде, где есть `isLoading` + `ProgressView`, привести к одному виду через общий компонент при рефакторинге.

## 4. Обработка ошибок

**Принцип:** Все ошибки при работе с сервером обрабатываются и доводятся до пользователя в понятном виде. Так как текущий бэкенд (Firebase) не отдаёт структурированные коды ошибок, приложение само определяет тип ошибки (по сообщению или коду) и показывает текст из своего словаря/маппинга.

### 4.1 Что сделано

- **Словарь ошибок:** модуль `Core/Errors/AppErrors.swift` — функция `AppErrors.userMessage(for: Error)` возвращает пользовательское сообщение на русском. Учитываются: Firebase Auth (неверный пароль, нет сети, слишком много попыток и т.д.), Firestore (permission-denied, unavailable), URLError (нет сети, таймаут). Для остальных — «Что-то пошло не так. Попробуйте позже».
- **Использование:** во всех экранах при `catch` вызывать `errorMessage = AppErrors.userMessage(for: error)` (или `onError(AppErrors.userMessage(for: error))`), показывать через alert или общий компонент.
- **Глобальные ошибки (загрузка профилей, удаление профиля):** в `AppState` поле `globalError`; в `RootView` — один alert «Ошибка» с этим сообщением.
- **Отображение:** везде alert «Ошибка» с текстом из `AppErrors.userMessage(for:)`. Не использовать сырой `error.localizedDescription`.

### 4.2 Правила

- Любой `catch` при вызове сервиса должен либо показать пользователю сообщение из словаря, либо пробросить выше к месту, где это сообщение показывается.
- Не глотать ошибки без логирования/показа пользователю.

## 5. Чеклист для новых экранов и доработок

- [ ] Кнопки и блоки — из `Core/Design` (не кастомная разметка с нуля).
- [ ] Цвета/размеры/шрифты — из `AppDesign` или компонентов.
- [ ] Загрузка данных — есть состояние загрузки (лоадер), нет «пусто → потом данные без индикатора».
- [ ] Ошибки сервера — обрабатываются и показываются через единый словарь/компонент.
- [ ] Вызовы к данным — только через сервисы (протоколы), без прямой работы с Firebase во View.
